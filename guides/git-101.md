# Git 101 для курса «Инженерия Искусственного Интеллекта»

Этот файл — короткий вводный гайд по Git **под задачи курса**:

- работа со своим репозиторием по шаблону `aie-student-template`;
- сдача домашних в `homeworks/`;
- ведение сквозного проекта в `project/`.

Цель: чтобы ты уверенно делал базовые операции, не боялся Git и мог самостоятельно сдавать задания.

---

## 1. Что такое Git и репозиторий

**Git** — это система контроля версий. Она помогает:

- хранить историю изменений в коде и документах;
- откатываться к прошлым версиям;
- работать над одним проектом в команде.

**Репозиторий** — папка с проектом и скрытая папка `.git` внутри.  
Всё, что ты делаешь в рамках курсового репо, — это операции над этим репозиторием.

В курсе у тебя будет как минимум **один личный репозиторий**, созданный по шаблону `aie-student-template`.

---

## 2. Начальная настройка Git

Сделай это один раз на своей машине.

### 2.1. Установка

- Windows / macOS / Linux: установить Git с официального сайта или через пакетный менеджер.
- Проверка:

```bash
git --version
```

Если выводится версия (например, `git version 2.4x...`) — всё ок.

### 2.2. Имя и почта

Эти данные будут записываться в каждый коммит (изменение):

```bash
git config --global user.name "Имя Фамилия"
git config --global user.email "your_email@example.com"
```

Проверить:

```bash
git config --global --list
```

Важно: используй тот же email, который связан с твоим аккаунтом GitHub (чтобы коммиты правильно сопоставлялись с профилем).

## 3. Клонирование репозитория

Когда преподаватель даст ссылку на твой репозиторий (по шаблону `aie-student-template`):

1. Выбери папку, где хочешь хранить проекты.
2. В терминале/PowerShell перейди туда:

    ```bash
    cd путь/до/папки
    ```

3. Склонируй репозиторий по HTTPS или SSH (как будет удобнее):

    ```bash
    git clone https://github.com/ORG/REPO.git
    # или
    git clone git@github.com:ORG/REPO.git
    ```

4. Зайди внутрь:

```bash
cd REPO
```

Теперь это локальная копия твоего репозитория. Дальше всё происходит **из этой папки**.

## 4. Типичный рабочий цикл

Базовый паттерн работы с Git:

1. **посмотреть статус**;
2. **добавить изменения** в индекс;
3. **сделать коммит**;
4. **отправить на GitHub**.

### 4.1. Проверка статуса

```bash
git status
```

Git покажет:

- какие файлы изменены;
- какие не отслеживаются;
- какие уже подготовлены к коммиту.

### 4.2. Добавление изменений

Добавить конкретный файл:

```bash
git add path/to/file.py
```

Добавить все изменённые и новые файлы:

```bash
git add .
```

(Но лучше иногда быть осторожным и явно добавлять только нужные файлы.)

### 4.3. Коммит (фиксируем изменения)

```bash
git commit -m "Краткое описание изменений"
```

Сообщение должно быть **понятным через неделю**:

- плохо: `"fix"`, `"update"`, `"123"`;
- лучше: `"Добавил решение HW01"` или `"Исправил баг в функции загрузки данных"`.

### 4.4. Отправка на GitHub (push)

```bash
git push
```

По умолчанию это отправит изменения в основную ветку (обычно `main`).

Если Git просит логин/пароль:

- при HTTPS сейчас вместо пароля используется **Personal Access Token** (PAT);
- или можно настроить SSH-ключи и работать по SSH.

## 5. Обновление локального репо (pull)

Если были изменения на GitHub (например, ты правил код на другом компьютере):

```bash
git pull
```

Это подтянет последние коммиты с сервера в твою локальную копию.

Если Git пишет про конфликты — см. кратко в разделе 7.

---

## 6. Ветки и политика для курса

По умолчанию в курсе достаточно одной основной ветки — `main`.

- Можно создавать дополнительные ветки (например, для экспериментов), но **не обязательно**.
- Если ветки пугают — работай в `main`, главное:

  - часто коммить;
  - не держать незакоммиченные изменения неделями.

Для интересующихся:

```bash
git branch          # список веток
git branch new-branch
git checkout new-branch
git checkout main
```

Но это не «обязательная программа» курса.

## 7. Типичные проблемы и как их решать (минимум)

### 7.1. «Я что-то изменил, но Git ничего не видит»

Проверь, в той ли ты папке:

```bash
pwd     # Linux/macOS
cd      # Windows PowerShell покажет путь в приглашении
```

И запусти:

```bash
git status
```

Если видишь:

- `untracked files` — новые файлы, их надо добавить `git add`.
- `modified` — изменённые файлы, тоже добавить и закоммитить.

### 7.2. Конфликт при `git pull`

Простейший сценарий:

- ты изменил файл локально;
- тот же файл изменился на GitHub;
- Git не знает, как их слить автоматически.

Git помечает конфликтные места в файле специальными маркерами `<<<<<<<`, `=======`, `>>>>>>>`.

Минимально, что нужно сделать:

1. Открыть файл в редакторе.
2. Найти эти маркеры.
3. Оставить нужный вариант (или объединить оба), удалить маркеры.
4. Добавить файл и закоммитить:

```bash
git add конфликтный_файл.py
git commit -m "Разрешил конфликт при merge"
git push
```

Если страшно — **не паниковать**. Можно:

- скопировать свои важные изменения куда-нибудь;
- сделать `git reset --hard` (осторожно!):

  - вернёт репо к состоянию последнего коммита;
- аккуратно вернуть свои изменения уже поверх обновлённой версии.

Но в идеале — не доводить до конфликтов: регулярно делать `git pull` перед началом работы.

## 8. Git и сдача домашних / проекта

### 8.1. Домашние задания

Для домашки `HWNN`:

1. В своей локальной копии репо:

   - создать/заполнить папку `homeworks/HWNN/` по заданию;
2. Проверить статус:

    ```bash
    git status
    ```

3. Добавить файлы домашки:

    ```bash
    git add homeworks/HWNN/
    git commit -m "Домашка HWNN: описание"
    git push
    ```

4. Убедиться на GitHub (через веб-интерфейс), что нужные файлы **точно** лежат в `homeworks/HWNN/`.

### 8.2. Сквозной проект

Аналогично, но в папке `project/`:

- ноутбуки → `project/notebooks/`;
- код → `project/src/`;
- конфиги → `project/configs/`;
- отчёт → `project/report.md`.

Перед дедлайном проекта сделай отдельный осмысленный коммит:

```bash
git status
git add project/
git commit -m "Готовая версия сквозного проекта перед защитой"
git push
```

## 9. Мини-чеклист перед сдачей

Перед тем как считать, что домашка/проект «сданы»:

1. **Локально:**

   - `git status` — должно быть `nothing to commit, working tree clean`;
   - код/ноутбуки запускаются хотя бы в минимальном сценарии.

2. **Коммиты:**

   - есть осмысленный коммит под эту домашку/этап проекта;
   - не осталось временных файлов (`.ipynb_checkpoints/`, `.DS_Store`, огромных мусорных файлов и т.п.).

3. **GitHub:**

   - зайти в веб-интерфейс;
   - проверить, что в нужной ветке (обычно `main`) действительно есть:

     - нужные папки (`homeworks/HWNN/`, `project/`),
     - нужные файлы,
     - актуальная версия (дата/время коммита до дедлайна).

4. **Инструкции:**

   - есть короткое описание в `README` (или отдельном `README.md` в папке домашки/проекта), как запускать код;
   - если нужны данные, понятно, откуда их взять (репозиторий / скрипт загрузки / ссылка).

## 10. Если что-то пошло не так

Типичный порядок действий:

1. **Не паниковать.**
2. Посмотреть, что говорит `git status`.
3. Скопировать важные изменения (в крайнем случае) куда-нибудь отдельно.
4. При необходимости:

   - написать преподавателю/ассистенту;
   - приложить:

     - команду, которую запускал;
     - вывод `git status`;
     - краткое описание, что хочешь получить.

Хорошая новость: большинство «страшных» ситуаций Git можно исправить, главное — не усугублять их случайными командами.
