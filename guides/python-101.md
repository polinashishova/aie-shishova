# Python 101 для курса «Инженерия Искусственного Интеллекта»

Этот файл — короткий практический гайд по Python **под задачи курса**:

- работа в своём репозитории (`homeworks/` и `project/`);
- настройка виртуального окружения;
- установка библиотек;
- запуск скриптов и ноутбуков;
- базовая структура кода.

Цель: не «научить Python с нуля», а выровнять техническую базу так, чтобы можно было спокойно заниматься инженерией ИИ.

## 1. Версия Python и окружение

### 1.1. Версия Python

В курсе ориентируемся на современную версию Python 3.x (актуальные детали преподаватель укажет отдельно).  

Проверка:

```bash
python --version
# или
python3 --version
```

Если видишь что-то вроде `Python 3.10.x` или `3.11.x` — ок для курса.

> Важно: на одной машине может быть несколько Python. На *nix часто нужна именно команда `python3`, а не `python`.

## 2. Виртуальное окружение (venv)

**Виртуальное окружение** позволяет:

- изолировать зависимости проекта от системы;
- не ломать другие проекты, когда ставишь новые пакеты.

### 2.1. Создание окружения

В корне своего репозитория:

```bash
python -m venv .venv
# или (если python3)
python3 -m venv .venv
```

Появится папка `.venv` — её **не** нужно трогать руками и **нельзя** коммитить в Git.

### 2.2. Активация

**Windows (PowerShell):**

```powershell
.\.venv\Scripts\Activate.ps1
```

**Linux/macOS (bash/zsh):**

```bash
source .venv/bin/activate
```

Признаки, что venv активирован:

- в начале строки терминала появилось что-то вроде `(.venv)`;
- `python` и `pip` теперь работают **внутри** окружения.

### 2.3. Деактивация

Когда закончил:

```bash
deactivate
```

## 3. Установка пакетов и requirements.txt

### 3.1. Установка пакета

После активации venv:

```bash
pip install numpy
pip install pandas
pip install "scikit-learn"
```

Проверка, что пакет есть:

```bash
python -c "import numpy; print(numpy.__version__)"
```

### 3.2. `requirements.txt`

Файл `requirements.txt` описывает, какие пакеты нужны проекту.

Простой вариант создания:

```bash
pip freeze > requirements.txt
```

Но лучше **не** замораживать всё подряд, а указывать только основные зависимости, например:

```text
numpy
pandas
scikit-learn
matplotlib
jupyter
```

Установка зависимостей на новой машине:

```bash
pip install -r requirements.txt
```

> Идея курса: чтобы проверяющий мог сделать `git clone`, `python -m venv`, `pip install -r requirements.txt` и запустить твой код без танцев.

## 4. Запуск скриптов и модулей

### 4.1. Обычный скрипт

Есть файл `script.py`:

```bash
python script.py
# или
python3 script.py
```

Если скрипт лежит глубже:

```bash
python project/src/train_model.py
```

### 4.2. Запуск как модуль

Иногда удобно запускать модуль из пакета:

```bash
python -m project.src.train_model
```

Где:

- `project` — корневая папка пакета;
- `src` — подпапка;
- `train_model.py` — модуль.

## 5. Jupyter-ноутбуки

### 5.1. Установка

Внутри venv:

```bash
pip install jupyter
```

### 5.2. Запуск

В корне проекта:

```bash
jupyter notebook
```

Откроется браузер с интерфейсом Jupyter.
Можно заходить в `homeworks/HWNN/` или `project/notebooks/` и создавать/редактировать ноутбуки.

### 5.3. Рекомендации по ноутбукам

- Не держи весь проект в одном монструозном ноутбуке.
- Для **проектов**:

  - EDA → один/несколько ноутбуков;
  - эксперименты → отдельные ноутбуки;
  - итоговый код → выносить в `src/`.
- Перед сдачей:

  - убедись, что ноутбук можно выполнить «сверху вниз» без ошибок;
  - по возможности сделай `Kernel → Restart & Run All` и проверь.

## 6. Структура кода и импорт модулей

### 6.1. Минимальная структура под проект

Простой вариант (пример для `project/`):

```text
project/
  README.md
  report.md
  self-checklist.md
  requirements.txt
  notebooks/
  src/
    __init__.py
    data/
      __init__.py
      load_data.py
    models/
      __init__.py
      train.py
      predict.py
    service/
      __init__.py
      api.py
  data/
  configs/
  tests/
```

Файл `__init__.py` (можно пустой) говорит Python: «это пакет, отсюда можно импортировать».

### 6.2. Импорт своих модулей

Например, в `project/src/service/api.py`:

```python
from project.src.models.predict import load_model, predict_one
from project.src.data.load_data import preprocess_input
```

Чтобы такой импорт работал, важно:

- запускать код из корня проекта (где лежит папка `project/`);
- не переименовывать папки хаотично;
- не делать файлы с именами, совпадающими с именами модулей стандартной библиотеки (например, `random.py`, `json.py`).

## 7. Базовый стиль и хорошие привычки

### 7.1. Имена

- переменные и функции: `snake_case` (`train_model`, `load_data`, `user_id`);
- классы: `CamelCase` (`MyModel`, `DataLoader`);
- файлы и модули: тоже `snake_case` (`train_model.py`, `data_utils.py`).

### 7.2. «Главный блок» в скрипте

Если скрипт можно запускать и импортировать:

```python
def main():
    # основной код
    ...

if __name__ == "__main__":
    main()
```

Так:

- при запуске `python script.py` выполнится `main()`;
- при `import script` код из `main()` не будет запускаться автоматически.

### 7.3. Логи вместо `print` (по мере взросления проекта)

На старте `print` ок, но для проекта удобно использовать `logging`:

```python
import logging

logging.basicConfig(level=logging.INFO)

logging.info("Старт обучения")
logging.warning("Что-то странное с данными")
logging.error("Модель не загрузилась")
```

Это пригодится, когда будем говорить о наблюдаемости и сервисах.

## 8. Типичные ошибки и как их избегать

### 8.1. «ModuleNotFoundError: No module named ...»

Чаще всего:

- неправильный путь импорта (`from src import ...` вместо `from project.src import ...`);
- запуск скрипта не из корня репозитория;
- отсутствует `__init__.py` в папках, которые должны быть пакетами.

Стратегия:

1. Убедись, что стоишь в **корне** проекта (`pwd` / смотри путь).
2. Запусти через:

    ```bash
    python -m project.src.service.api
    ```

3. Добавь `__init__.py` в нужные папки.

---

### 8.2. «Отсутствующие библиотеки» (`ModuleNotFoundError` на numpy/pandas/... )

Проверка:

- активировано ли venv;
- установлен ли пакет (`pip list`).

Если нет — установить:

```bash
pip install numpy
# и затем обновить requirements.txt (по необходимости)
```

---

### 8.3. Жёстко зашитые пути

Плохо:

```python
df = pd.read_csv("C:\\Users\\Имя\\Desktop\\data.csv")
```

Лучше:

```python
from pathlib import Path

DATA_DIR = Path(__file__).parent.parent.parent / "data"
df = pd.read_csv(DATA_DIR / "data.csv")
```

Так код будет работать и на твоей машине, и у проверяющего после `git clone`.

## 9. Мини-чеклист перед тем, как «всё готово»

Перед тем как считать, что твоя домашка/часть проекта **готова технически**:

1. В корне проекта:

   - активировать venv;
   - запустить установку зависимостей:

     ```bash
     pip install -r requirements.txt
     ```

2. Убедиться, что:

   - основной скрипт/ноутбуки запускаются без ошибок;
   - пути к файлам относительные, а не «C:\…».
3. Проверить:

   - `python --version` (та, с которой ты всё делал);
   - `git status` (нет незакоммиченных критически важных файлов).

Если всё это выполняется — ты в хорошем состоянии для сдачи.

## 10. Если Python «ломается»

Если что-то пошло совсем не так:

  1. Не пугаться.
  2. В крайнем случае:

     - удалить папку `.venv`;
     - создать её заново;
     - снова установить зависимости:

     ```bash
     python -m venv .venv
     source .venv/bin/activate  # или .\.venv\Scripts\Activate.ps1
     pip install -r requirements.txt
     ```

  3. Проверить простейший скрипт:

```python
print("Python жив и здоров")
```

Если он запускается — значит, базовая инфраструктура в порядке, дальше можно разбираться с логикой проекта.
