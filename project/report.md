## 1. Паспорт проекта

- **Название проекта:** `Гибридный рекомендательный API сервис фильмов с контент-базированной моделью и колаборативной фильтрацией`
- **Автор:** `Шишова Полина Сергеевна`

               почта:    `timewastingnonsense@gmail.com`
- **Ссылка на репозиторий:** `https://github.com/polinashishova/aie-shishova/`


Проект реализует гибридный сервис рекомендаций фильмов, объединяющий контент-базированный подход (на основе жанров, тегов и года выпуска) и коллаборативную фильтрацию (на основе оценок пользователей).
Используется открытый датасет, простые content-based модели, модели коллаборативной фильтрации и случайные baseline-модели.
Результат — REST API, возвращающий для запрошенного ID фильма список похожих, а для ID пользователя — персонализированные рекомендации.

## 2. Постановка задачи и контекст

### 2.1. Предметная область и задача:

   Предметная область: рекомендательные системы в сфере киноиндустрии и видеосервисов (OTT-платформы, онлайн-кинотеатры). 

   Решаемая задача: построение гибридного сервиса рекомендаций фильмов, который способен:
   - рекомендовать фильмы новым пользователям (user cold-start) на основе похожести контента;
   - предоставлять персонализированные рекомендации пользователям с историей оценок на основе их вкусов и предпочтений.
   
   Потенциальные пользователи: зрители онлайн-кинотеатров, которые хотят найти фильмы для просмотра.
   
   Сценарий использования: пользователь заходит на платформу, и система предлагает ему подборку фильмов, которые могут ему понравиться; 
   - если пользователь новый, система рекомендует популярные или похожие на выбранный фильмы; 
   - если у пользователя есть история просмотров/оценок, система анализирует его вкусы и предлагает персонализированные рекомендации.
  
### 2.2. Формулировка задачи в терминах ML/ИИ:
   
   Входные данные:
   - рейтинги пользователей - оценки фильмов конкретными пользователями (0.5 - 5.0);
   - метаданные фильмов - названия и жанры;
   - теги фильмов, оставленные пользователями.
   
   Гибридная рекомендательная система объединяет:
   - контент-базированную фильтрацию (задача поиска похожих фильмов);
   - коллаборативную фильтрацию (Alternating Least Squares для неявных откликов - задача персонализированных рекомендаций).
   
   Целевая пееременная / выход модели:
   - для контент-базированной модели - список k фильмов с рассчитанной косинусной близостью относительно запрошенного фильма;
   - для коллаборативной фильтрации - список k фильмов, которые с наибольшей вероятностью понравятся пользователю, с предсказанными скорами релевантности.
   
   Среди ограничений и требований - скорость ответа.

### 2.3. Целевые метрики качества:
   
   В обоих случаях решается задача ранжирования, а не точного предсказания рейтинга, поэтому для оценки качества моделей используются метрики качества ранжирования.

   | Метрика | Формула | Почему выбрана |
   |---------|---------|----------------|
   | **Precision@K** | `#релевантных_в_топ-K / K` | Показывает, какая доля рекомендованных фильмов действительно релевантна |
   | **Recall@K** | `#релевантных_в_топ-K / #всех_релевантных` | Важно, чтобы рекомендации покрывали большую часть интересов пользователя |
   | **Hit Rate@K** | `1 если есть хотя бы 1 релевантный в топ-K, иначе 0` | Пользователю важно получить хотя бы один интересный фильм |
   | **NDCG@K** | `DCG / IDCG` | Учитывает порядок рекомендаций (чем выше релевантность, тем выше позиция) |

   **Почему эти метрики:**
   - В рекомендательных системах пользователю важнее, чтобы интересные фильмы были **среди первых** предложенных, чем их точный порядок.
   - **Recall@K** важен, так как если система не покажет ни одного интересного фильма, пользователь может уйти с платформы.
   - **Hit Rate@K** критичен для бизнеса — чем чаще пользователь получает хотя бы одну релевантную рекомендацию, тем выше вовлеченность.
   - **NDCG@K** штрафует алгоритм, если релевантный фильм оказывается в конце списка.

   **Базовое значение (baseline):** Случайная модель (precision@10 ≈ 0.05%, recall@10 ≈ 0.02%, hit_rate@10 ≈ 0.46%) для сравнения качества обученных моделей.

## 4. Модели и подходы

В датасете представлены оценки от 0.5 до 5.0, не все из них равноначны и так как решается задача ранжирования, а не точного предсказания рейтинга, наличие негативных или нейтральных оценок (0.5-3.0) может может влиять на качество. Поэтому для экмпериментов с выбором контент-базированной модели было выделено три выборки: все рейтинги, не негативные (3.0-5.0), положительные (4.0-5.0).

Более того, все ранжирующие метрики считались с k=10. Такое число было выбрано как стандартный компромисс между полнотой рекомендаций и их релевантностью.

   - в качестве baseline (`notebooks/02_preprocessing_and experimets.ipynb` раздел 2.3.3. Оценка baseline) для обоих сценариев использовались случайные модели рекомендаций: алгоритм выбирал k рандомных фильмов из обучающей выборки;
   - результаты получились следующими:
  
      |подвыборка  | precision@10 | recall@10     |hit_rate@10 | ndcg@10|

      |все рейтинги|0.000461	    |0.000137	     |0.004558	   |0.000350|

### 4.2. Улучшенные модели и эксперименты:

   - для рекомендаций похожих фильмов (`notebooks/02_preprocessing_and experimets.ipynb` раздел 2.3.4. Обучение и оценка content-based моделей рекомендаций) в контент-базированном варианте сравнивались модели обученные на различных вариациях признаков:
     - one-hot жанры;
     - TF-IDF обученный на жанрах + десятилетиях;
     - TF-IDF обученный на жанрах + десятилетиях + тегах + понижение размерности с помощью TruncatedSVD с сохранением 0.95 дисперсии данных (на всех трех выборках осталась лишь одна компонента);
     - TF-IDF обученный на жанрах + десятилетиях + тегах + понижение размерности с помощью TruncatedSVD с сохранением 100 компонент;
   - в качестве профиля пользователя использовался последний наиболее высоко оцененный пользователем фильм из обучающей выборки;
   - для выбора модели коллаборативной фильтрации эксперименты проводились только на выборке положительных рейтингов с обучением модели Alternating Least Squares из библиотеки implicit для обучения эмбеддингов пользователей и фильмов на неявных откликах (`notebooks/02_preprocessing_and experimets.ipynb` раздел 2.3.5. Обучение и оценка collaborative filtering моделей рекомендаций);
   - все рейтинги (4.0-5.0) были заменены на 1.0; негативные и нейтральные оценки, а также те фильмы, с которыми пользователь не взаимодействовал, были отмечены нулями в матрице (пользователи, фильмы);
   - решение отказаться от использования явных оценок для обучения модели коллаборативной фильтрации обусловлено тем, что для решения поставленной задачи, важнее в целом рекомендовать пользователю те фильмы, которые могут ему понравиться;
   - эксперименты с ALS моделью заключаются в переборе таких гиперпараметров, как размерность эмбеддингов, сила регуляризации, количество итераций, вес положительных примеров;
   - дополнительно, во всех эспериментах матрица user-item взвешена по схеме BM25.

Все эксперименты, их подробное описание и их результаты представлены в ноутбуке `notebooks/02_preprocessing_and experimets.ipynb`.

## 5. Экспериментальный протокол и результаты


### 5.1. Экспериментальный протокол

**Разделение данных:**
- Для контент-базированных экспериментов использовались три варианта выборок:
  - **Все рейтинги** (0.5-5.0) — 32 млн записей
  - **Не негативные** (3.0-5.0) — 26.3 млн записей
  - **Положительные** (4.0-5.0) — 15.9 млн записей
- Для коллаборативной фильтрации использовались только положительные рейтинги (4.0-5.0), бинаризованные до 1.
- Каждая выборка разделена на train/test **временным сплитом** по пользователям (80/20).
- Пользователи с менее чем 5 оценками отфильтрованы.

**Протокол для content-based моделей:**
1. Профиль пользователя = последний наиболее высоко оцененный фильм из train.
2. Вычисление косинусной близости с фильмами из test.
3. Выбор top-10 фильмов по близости.
4. Ground truth = все фильмы, оцененные пользователем в test.
5. Расчет метрик ранжирования.

**Протокол для collaborative filtering моделей:**
1. Построение матрицы user-item (пользователи × фильмы) с весами взаимодействий.
2. BM25 взвешивание (K1=100, B=0.8).
3. Обучение ALS с различными гиперпараметрами.
4. Получение top-10 рекомендаций через встроенный метод `recommend()`.
5. Исключение негативных фильмов (оценки < 4.0).
6. Расчет метрик относительно тестовых взаимодействий.

### 5.3. Выбор финальной модели

**Content-based модель:**
- **Выбрана:** `nonneg_ratings_tfidf_genres_decade_tags_svd_without_variance_ratio`
- **Выборка:** Не негативные рейтинги (3.0-5.0)
- **Параметры:** TF-IDF + TruncatedSVD (n_components=100)
- **Метрики:** precision@10=0.0150, recall@10=0.0103, hit_rate@10=0.1226, ndcg@10=0.0146
- **Почему:** Лучший баланс между hit_rate (12.3%) и recall (1.03%). Для cold-start сценария важно, чтобы пользователь получил хотя бы один интересный фильм (hit_rate), при этом сохранение recall позволяет покрыть разные интересы.

**Collaborative filtering модель:**
- **Выбрана:** ALS с параметрами `factors=100, regularization=0.1, iterations=50, alpha=1.0`
- **Метрики:** precision@10=0.0886, recall@10=0.0909, hit_rate@10=0.5121, ndcg@10=0.1133
- **Почему:** Наилучшие показатели по всем метрикам — более половины пользователей (51.2%) получают хотя бы одну релевантную рекомендацию в топ-10. Увеличение iterations до 100 не дает значимого улучшения, поэтому выбрано 50 итераций для оптимальной скорости обучения.

**Ключевые trade-off’ы:**
- Размерность эмбеддингов (100 vs 50): качество улучшается на ~1%, но требует больше памяти и времени для обучения
- Регуляризация (0.1 vs 0.01): более высокая регуляризация немного снижает качество, но улучшает обобщение
- Параметр alpha (1.0 vs 10.0): увеличение ухудшает все метрики, поэтому оставлен стандартный

### 6.2. API и endpoints

|----------|-------|----------|-----------|-------|

| `/predict` | POST | Получение рекомендаций | `movieIds` или `userIds`, `k` (1-100) | `{recommendations, model_used, timestamp, request_id}` |

| `/redoc` | GET | Redoc | - | HTML-документация |

```json
  "k": 10
}
```

## 7. Наблюдаемость, конфигурация и безопасность

### 7.1. Логи и наблюдаемость

**Что логируется:**
- Запросы к API (ID запроса, k, количество movieIds/userIds).
- Загрузка данных в память, состояние обработки данных, сохранение данных (если запускается соответствующий скрипт).
- Загрузка моделей в память (успех/ошибка).
- Health check проверки.
- Ошибки валидации и обработки запросов.
- Очистка кэша при остановке сервера.

**Health check endpoint:** `/health` возвращает статус сервиса и флаг загрузки моделей. Используется в Docker для мониторинга состояния контейнера.

### 7.2. Конфигурации

| Файл | Назначение | Параметры |
|------|------------|-----------|
| `configs/paths.json` | Пути к файлам и директориям | `data_raw_dir`, `models_dir`, `artifacts_dir` и др. |
| `configs/models.json` | Гиперпараметры моделей | `svd.n_components`, `als.factors`, `als.regularization` |
| `.env.example` | Переменные окружения | `APP_HOST`, `APP_PORT`, `LOG_LEVEL`, пороги рейтингов |

**Что можно перенастроить без изменения кода:**
- Хост и порт сервера.
- Уровень логирования.
- Пороги рейтингов для CB и CF моделей.
- Колонка для текстовых признаков CB модели.
- Путь к конфигурационным файлам, названия конечных файлов.

### 7.3. Безопасность

**Принятые меры:**
- Токены и секреты не хранятся в коде, в данном проекте они и не были нужны.
- В репозитории использован `.env.example` вместо реального `.env`.
- `.gitignore` исключает `.env`, `artifacts/logs/`, `__pycache__/`.
- Валидация входных данных через Pydantic (ограничение длины списков, проверка положительности ID).
- Защита от path traversal в `extract_archive()`.
- Непривилегированный пользователь в Docker (`appuser`)

**Известные ограничения:**
- ID пользователей логируются в открытом виде (данные публичные, ID анонимизированы).
- Маппинги ID хранятся в открытых JSON-файлах.
- CORS открыт для всех источников.

## 8. Ограничения и дальнейшая работа

### 8.1. Текущие ограничения

| Категория | Ограничение | Пояснение |
|-----------|-------------|-----------|
| **Данные** | Только MovieLens 32M | Для других источников нужна адаптация |
| **Обучение** | Offline, нет онлайн-обновления | Новые оценки не влияют на рекомендации |
| **Каталог** | Новые фильмы не добавляются | Только фильмы из обучающей выборки |
| **Нагрузка** | Нет горизонтального масштабирования | При высоких нагрузках может не справиться |
| **Безопасность** | Нет rate limiting | Уязвим для DoS-атак |
| **Кэширование** | Одинаковые запросы не кэшируются | Избыточные вычисления |
| **Интерпретация** | Нет объяснений рекомендаций | Пользователь не понимает "почему" |
|**Конфиденциальность**| ID пользователей логируются напрямую| Злоумышленник может воспользоваться информацией о пользователе |


### 8.2. Дальнейшая работа

Если бы было больше времени, я бы уделила больше внимания следующим вещам:

**Улучшения моделей:**
