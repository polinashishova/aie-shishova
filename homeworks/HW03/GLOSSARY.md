# Глоссарий курса «Инженерия Искусственного Интеллекта»

Этот файл — справочник по терминам, которые используются в лекциях, семинарах, домашних заданиях и проекте.

- Определения даны **на уровне инженерного курса**, а не строго академически.
- Английские термины указаны там, где они реально используются в индустрии.
- Глоссарий не претендует на полноту, но охватывает ключевые слова, которые будут встречаться по ходу курса.

---

## 1. Роли и общие понятия

- **ИИ-система** *(AI system)* — программно-аппаратный комплекс, в котором есть один или несколько компонентов на основе моделей машинного обучения и вокруг них — обычный код, данные, интерфейсы и инфраструктура.

- **ИИ-сервис** *(AI service)* — сервис (обычно веб- или API-сервис), предоставляющий функции ИИ-системы наружу: `POST /predict`, чат-бот, интерфейс аналитика и т.п.

- **Инженерия ИИ** *(AI engineering)* — набор практик, как **проектировать, разрабатывать, запускать и сопровождать** ИИ-системы как продукт, а не только тренировать модели.

- **Data Scientist (DS)** — специалист по постановке задач на данных, выбору и обучению моделей, анализу результатов и экспериментов.

- **ML Engineer (MLE)** — инженер, отвечающий за реализацию и эксплуатацию моделей в коде и сервисах: пайплайны, деплой, производительность, интеграции.

- **MLOps Engineer** — инженер, отвечающий за инфраструктуру для ML: пайплайны данных и экспериментов, деплой моделей, мониторинг, реестры моделей и т.п.

- **AI Engineer / Applied ML Engineer** — «скрещённый» профиль, совмещающий часть DS, MLE и классической software engineering (то, куда целится этот курс).

- **Жизненный цикл ИИ-системы** *(AI lifecycle)* — последовательность этапов: постановка задачи → данные → эксперименты → модель → сервис/деплой → мониторинг → итерации/обновления.

- **Артефакт** *(artifact)* — любой «побочный продукт» разработки и эксплуатации: датасеты, модели, конфиги, логи, отчёты, ноутбуки, скрипты, Docker-образы.

## 2. Данные и датасеты

- **Датасет** *(dataset)* — набор данных с более-менее общей структурой, используемый для обучения и оценки моделей.

- **Объект / запись / пример** *(sample / instance)* — одна строка в табличном датасете, один документ, одно изображение и т.п.

- **Признак** *(feature)* — отдельная измеримая характеристика объекта (столбец в табличных данных, признак из текста/изображения и т.п.).

- **Целевая переменная** *(target / label)* — то, что модель пытается предсказать (класс, вероятность, число, ранжирование).

- **Train / validation / test** — разбиения датасета:
  - **train** — для обучения модели;
  - **validation (val)** — для подбора гиперпараметров, выбора модели;
  - **test** — для финальной оценки, которая не должна «подсматриваться» в процессе настройки.

- **Кросс-валидация** *(cross-validation)* — техника оценки модели, когда датасет многократно разбивается на обучающую и валидационную части разными способами.

- **Утечка данных** *(data leakage)* — ситуация, когда информация из будущего/валидации/теста «подглядывает» в обучение, и модель показывает нереалистично высокие результаты.

- **Дрейф данных / концепции** *(data / concept drift)* — изменение распределения входных данных или зависимости «вход → цель» во времени, из-за чего качество модели в продакшене падает.

- **Фича-инженерия** *(feature engineering)* — процесс конструирования и выбора признаков из исходных данных.

- **Анонимизация / обезличивание** — удаление или замена персональных идентификаторов, чтобы нельзя было восстановить конкретного человека или объект.

## 3. Модели, обучение и метрики

- **Модель** *(model)* — функция, которая по входным данным выдаёт предсказание (класс, число, текст и т.п.), плюс набор параметров (веса).

- **Бейзлайн** *(baseline)* — простое, понятное решение, с которым сравниваются более сложные модели (часто rule-based или простая модель).

- **Гиперпараметры** *(hyperparameters)* — параметры обучения и структуры модели, которые не учатся автоматически (глубина дерева, число деревьев, learning rate и т.п.).

- **Переобучение** *(overfitting)* — модель слишком хорошо подстраивается под обучающие данные и плохо обобщает на новые.

- **Недообучение** *(underfitting)* — модель слишком простая, не выучивает даже базовые закономерности.

- **Регуляризация** *(regularization)* — приём для борьбы с переобучением, уменьшающий «свободу» модели (штрафы на веса, ограничения сложности и т.п.).

- **Метрика качества** *(metric)* — численная характеристика того, насколько хорошо модель решает задачу (accuracy, ROC-AUC, RMSE и др.).

- **Accuracy** — доля правильных предсказаний среди всех.

- **Precision / Recall / F1** — метрики для задач классификации, особенно с несбалансированными классами:
  - **precision** — насколько предсказания «позитив» действительно позитивны;
  - **recall** — насколько хорошо мы находим все настоящие позитивы;
  - **F1** — гармоническое среднее precision и recall.

- **ROC-AUC** — метрика для бинарной классификации, измеряющая качество ранжирования по вероятности класса.

- **RMSE / MAE** — стандартные метрики для регрессии (корень из среднеквадратичной ошибки, средняя абсолютная ошибка).

- **Seed / random_state** — фиксированное начальное значение для генераторов случайных чисел, позволяющее повторить эксперимент.

- **Воспроизводимость** *(reproducibility)* — возможность запустить код на тех же данных и получить те же (или очень близкие) результаты.

## 4. Сервисы, API и архитектура

- **API** *(Application Programming Interface)* — программный интерфейс, через который внешний мир общается с сервисом (часто HTTP-API).

- **REST API** — стиль API, где операции делаются через HTTP-методы (GET, POST, PUT, DELETE) и понятные URL.

- **Endpoint** — конкретный маршрут и метод API, например `POST /predict` или `GET /health`.

- **Request / response** — запрос к API и ответ сервиса (обычно в формате JSON).

- **JSON** — текстовый формат передачи структурированных данных в веб-сервисах.

- **Inference** — использование уже обученной модели для предсказаний (в отличие от обучения).

- **Latency / задержка** — время между запросом и ответом сервиса.

- **Throughput / пропускная способность** — сколько запросов сервис способен обработать за единицу времени.

- **SLA / SLO** — договорённости и цели по доступности и качеству сервиса (uptime, задержки, ошибки).

- **Монолит / микросервисы** — варианты архитектуры:
  - монолит — «один большой» сервис;
  - микросервисы — набор небольших сервисов, каждый отвечает за свою функцию.

## 5. MLOps, пайплайны и инфраструктура

- **MLOps** — набор практик и инструментов для управления жизненным циклом моделей: данные, обучение, деплой, мониторинг, обновления, воспроизводимость.

- **Пайплайн** *(pipeline)* — последовательность шагов обработки: данные → предобработка → обучение → оценка → деплой.

- **CI/CD** *(Continuous Integration / Continuous Delivery/Deployment)* — автоматические процессы, которые:
  - проверяют код и запускают тесты при каждом изменении;
  - помогают безопасно и часто выкатывать новые версии.

- **Experiment tracking** — систематический учёт ML-экспериментов (конфиги, метрики, артефакты, версии данных и моделей).

- **Версионирование данных** *(data versioning)* — управление версиями датасетов и производных артефактов, чтобы знать, на каких данных обучалась конкретная модель.

- **Реестр моделей** *(model registry)* — система, где хранятся версии моделей и их статусы (черновик, staging, production, архив).

- **Docker / контейнер** — способ упаковать приложение с зависимостями в образ, который можно одинаково запускать на разных машинах.

- **Образ (image)** — шаблон для контейнера (содержит файловую систему и настройки запуска).

- **Том (volume)** — механизм Docker для хранения данных вне жизненного цикла контейнера.

## 6. Git, репозитории и проектная работа

- **Git** — система контроля версий для кода и текстовых файлов.

- **Репозиторий** *(repo)* — папка с проектом, в которой Git хранит историю изменений.

- **Commit** — «снимок» изменений в репозитории с сообщением (комментарием).

- **Ветка** *(branch)* — независимая линия разработки в репозитории.

- **Pull request (PR)** — предложение изменить код в репозитории, которое можно обсудить и принять/отклонить.

- **GitHub** — популярный хостинг Git-репозиториев с веб-интерфейсом, issue tracker’ом, CI и т.п.

- **Git LFS** *(Large File Storage)* — расширение Git для хранения больших бинарных файлов (веса моделей, крупные артефакты) без раздувания репозитория.

Подробные «как делать» живут в `git-101.md`, здесь — только термины.

## 7. LLM, RAG, агенты и мультимодальные системы

- **LLM** *(Large Language Model)* — большая языковая модель, обученная предсказывать следующий токен/слово по контексту, способная решать широкий спектр текстовых задач.

- **Токен** *(token)* — минимальная единица текста для модели (фрагмент слова, слово, знак).

- **Токенизатор** *(tokenizer)* — алгоритм, который превращает текст в последовательность токенов и обратно.

- **Эмбеддинг** *(embedding)* — числовой вектор, представляющий объект (слово, документ, картинку) в непрерывном пространстве, где близость отражает семантическую похожесть.

- **RAG** *(Retrieval-Augmented Generation)* — архитектура, где LLM сначала получает релевантные документы из хранилища (retrieval), а потом генерирует ответ с опорой на них (augmented generation).

- **Векторное хранилище** *(vector store)* — место, где хранятся эмбеддинги и по ним можно искать ближайшие векторы (по косинусному расстоянию и т.п.).

- **Агент** *(agent)* — система на основе LLM, которая может декомпозировать задачу, вызывать инструменты (API, базы данных, код) и использовать результаты в дальнейших шагах.

- **Инструмент / tool** — внешняя функция/сервис, к которому агент может обращаться (например, «поисковый API», «выполнить Python-код»).

## 8. Безопасность, доверие и эксплуатация ИИ

- **PII / Персональные данные** — данные, по которым можно идентифицировать человека (ФИО, телефоны, e-mail, адреса, документы и т.п.).

- **Секреты** *(secrets)* — чувствительная информация: пароли, токены, API-ключи, приватные ключи.

- **Переменная окружения** *(environment variable)* — способ передать конфигурацию/секреты приложению извне, не хардкодя их в коде.

- **Threat model / модель угроз** — описание того, какие угрозы безопасности для системы считаются реалистичными и что мы с ними делаем.

- **Мониторинг качества модели** *(model monitoring)* — отслеживание метрик качества, дрейфа данных и поведения модели в проде.

- **Наблюдаемость** *(observability)* — способность понять состояние системы через метрики, логи и трассировки.

- **Prompt injection** — класс атак на LLM-системы, когда злоумышленник через ввод подсовывает инструкции, меняющие поведение модели (игнорирование системных правил, утечка конфиденциальных данных и т.п.).

- **Trust / доверие к ИИ** — совокупность свойств и практик (качество, прозрачность, надёжность, безопасность, соответствие требованиям), из-за которых систему можно использовать в реальных задачах.
