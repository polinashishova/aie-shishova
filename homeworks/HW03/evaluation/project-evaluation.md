# Оценивание сквозного проекта

Этот документ описывает, **как оценивается сквозной мини-проект** по курсу «Инженерия Искусственного Интеллекта».

Для студентов основным рабочим инструментом является чеклист `project/self-checklist.md` и отчёт `project/report.md` в их репозитории.

---

## 1. Общие принципы

Итоговая оценка за проект выносится по пятибалльной шкале (2–5) и зависит от:

- выполнения **минимальных требований** к работоспособности сервиса и структуре проекта;
- количества выполненных пунктов чеклиста `project/self-checklist.md`;
- качества отчёта `project/report.md` и документации (`project/README.md`);
- аккуратности реализации, работы с данными и секретами.

Чеклист даёт ориентировочную самооценку, но окончательное решение остаётся за преподавателем.

---

## 2. Минимальные требования к принятию проекта

До того, как думать про баллы чеклиста, проект должен пройти **базовый порог**. Если он не пройден — оценка не может быть выше **2**.

Если сервис не запускается, `/predict` не использует модель или есть грубые нарушения по данным/секретам/плагиату — это основание для оценки **2** независимо от чеклиста.

---

## 3. Чеклист `project/self-checklist.md` и баллы

В студенческом репозитории есть файл `project/self-checklist.md` с десятью критериями. Каждый критерий оценивается как выполненный (`✅`) или нет (`❌`).

Критерии (в пересказе):

1. **Запуск сервиса**
   Сервис запускается по инструкциям из `project/README.md` и работает (минимум один рабочий сценарий).

2. **Реальная модель в `/predict`**
   Основной endpoint использует реальную обученную модель, а не заглушку.

3. **EDA и хотя бы один эксперимент**
   Есть ноутбук(и) с разведочным анализом данных и первым экспериментом с метриками.

4. **Сравнение моделей по метрикам**
   Есть хотя бы одно осмысленное сравнение нескольких моделей/конфигураций по метрикам.

5. **Структура кода**
   Код не свален в один ноутбук: есть внятная структура модулей в `src/` (например, `src/data/`, `src/models/`, `src/service/`).

6. **Развёртывание**
   Есть либо Dockerfile, либо чёткий сценарий развёртывания без Docker, описанный в `README.md`.

7. **Конфиги и секреты**
   Есть `.env.example`/конфиги, и при этом в репозитории нет реальных секретов/паролей.

8. **Наблюдаемость**
   Реализованы логи и базовая наблюдаемость (хотя бы консольные логи и endpoint `/health` или аналог).

9. **Обоснованный выбор финальной модели**
   В `report.md` есть раздел, где по результатам экспериментов обосновывается выбор финальной модели.

10. **Понятный сценарий демонстрации**
    `project/README.md` и `report.md` позволяют понять, как именно демонстрировать проект (что запускать, какие сценарии показывать).

**Подсчёт баллов:**

- Каждый честно выполненный критерий (`✅`) = **1 балл**.
- Максимум — **10** баллов.

Преподаватель при проверке может скорректировать отметки по критериям, если фактическое состояние проекта не соответствует самооценке.

---

## 4. Перевод баллов в итоговую оценку

Если минимальные требования (раздел 2) **не выполнены**, то итоговая оценка:

- **2** — проект не принят.

Если минимум выполнен, то ориентировочно:

- **1–4 балла по чеклисту** → оценка **3**
  - Проект реализован на базовом уровне:
    - минимальный функционал есть;
    - эксперименты и структура кода/документации — на уровне «лишь бы работало»;
    - многие элементы (наблюдаемость, работ с конфигами и секретами, чистота структуры) либо отсутствуют, либо сделаны минимально.

- **5–8 баллов по чеклисту** → оценка **4**
  - Хороший, рабочий проект:
    - сервис запускается, `/predict` использует модель;
    - есть EDA и серия экспериментов с метриками;
    - структура кода и конфигов в целом адекватна;
    - хотя бы часть инфраструктурных аспектов (Docker/наблюдаемость/секреты) реализована по требованиям курса.

- **9–10 баллов по чеклисту** → оценка **5**
  - Сильный, хорошо проработанный проект:
    - аккуратно реализован сервис и пайплайн;
    - проведены осмысленные эксперименты и чётко обоснован выбор финальной модели;
    - есть базовая наблюдаемость и грамотная работа с конфигами/секретами;
    - документация позволяет быстро понять и воспроизвести решение;
    - видно, что студент осмысленно прошёл все ключевые этапы из описания проекта.

Числа «1–4 / 5–8 / 9–10» — **ориентиp**, а не жёсткий критерий. В пограничных случаях преподаватель может принять решение в ту или иную сторону.

---

## 5. Дополнительные факторы

Помимо чеклиста, преподаватель может учитывать:

- качество кода (читаемость, простота, разумное разделение на модули);
- аккуратность и честность отчёта (`report.md`), ясность объяснения решений;
- дополнительные сильные стороны:
  - нестандартная постановка задачи;
  - продвинутые модели или архитектуры (RAG, агенты, мультимодальность), реализованные аккуратно;
  - осмысленные MLOps-элементы сверх минимума курса;
- соблюдение дедлайнов и договорённостей по этапам.

Эти факторы могут:

- поднять оценку на пол-ступени («верхняя» граница диапазона);
- либо, при серьёзных проблемах (например, неряшливая работа с данными/секретами, откровенный плагиат), наоборот, снизить её.

---

## 6. Как использовать этот документ

**Для студента:**

- начать с чтения `project/overview.md` и выбора темы;
- по мере работы периодически проходить по чеклисту `project/self-checklist.md`;
- перед сдачей проекта посмотреть на этот файл и честно оценить, на какой диапазон оценки проект тянет.

**Для преподавателя:**

- ориентироваться на минимальные требования (раздел 2) как на «проходной барьер»;
- использовать чеклист и таблицу соответствия баллов оценке как основу для прозрачности;
- фиксировать нетипичные решения и обстоятельства в комментариях к защите, чтобы студент понимал логику выставленной оценки.
