# Примеры тем для сквозного проекта

Этот документ помогает выбрать тему для сквозного мини-проекта.
Это **не список обязательных вариантов**, а ориентиры — можно брать любую близкую по духу задачу, согласовав её с преподавателем.

Условные пометки уровня:

- `[BASIC]` — базовый уровень, лего делать в одиночку с акцентом на классический ML.
- `[INTERMEDIATE]` — средний уровень, разумный баланс между моделью и сервисом.
- `[ADVANCED]` — продвинутый уровень, больше акцент на архитектуру, MLOps, LLM/RAG, агентов или мультимодальность.

---

## 1. Общие требования к теме

Независимо от домена, хорошая тема проекта:

1. **Решает осмысленную задачу**: понятный «пользователь» и его боль/метрика.
2. **Имеет доступные данные**:
   - открытые датасеты, либо
   - синтетические/обезличенные выборки, либо
   - данные, специально подготовленные для курса.
3. **Позволяет собрать полный цикл**:
   - постановка задачи → данные → модели → сервис → наблюдаемость → безопасность.
4. **Реалистична по объёму**:
   - за один курс реально довести до демо-сервиса,
   - без необходимости поднимать тяжёлые кластеры и терабайты данных.

## 2. Табличные бизнес-задачи

### 2.1. Кредитный/риск-скоринг по условному датасету `[INTERMEDIATE]`

**Идея:** сервис, который оценивает риск невозврата «кредита»/долга по профилю клиента или заявки.

- Вход: признаки клиента/заявки (доход, стаж, история, параметры заявки и т.п.).
- Выход: вероятность дефолта и категория риска (низкий/средний/высокий).
- Модели: логистическая регрессия, деревья/ансамбли, возможно простая нейросетка.
- Сервис: API, которому можно отправить заявку и получить оценку риска.

**Акценты:** интерпретируемость, выбор метрик, калибровка вероятностей.

---

### 2.2. Прогноз оттока клиентов (churn) `[INTERMEDIATE]`

**Идея:** сервис, который прогнозирует вероятность ухода пользователя/клиента.

- Вход: история взаимодействий клиента (покупки, визиты, активности).
- Выход: вероятность оттока в ближайший период.
- Модели: ансамбли, GBM, возможно простая нейросетка по агрегированным фичам.
- Сервис: API для получения вероятности ухода по профилю клиента.

**Акценты:** баланс классов, бизнес-интерпретация метрик (precision/recall, ROI от кампаний).

---

### 2.3. Оценка вероятности успешной сделки/конверсии `[BASIC]`

**Идея:** оценка вероятности, что лид/заявка дойдут до успешной сделки.

- Вход: признаки лида (источник, каналы, характеристики, поведение).
- Выход: вероятность успешной конверсии.
- Модели: классические ML-модели.
- Сервис: API для скоринга лида.

**Акценты:** простая архитектура, аккуратный экспериментальный дизайн.

---

### 2.4. Предсказание стоимости/спроса `[INTERMEDIATE]`

**Идея:** сервис, который прогнозирует цену или спрос (например, по характеристикам товара и рынку).

- Вход: характеристики товара, исторические данные.
- Выход: прогнозная цена/объём продаж.
- Модели: регрессия, ансамбли, возможно простая нейросетка.
- Сервис: API с предсказанием и доверительными интервалами (по возможности).

**Акценты:** регрессионные метрики (MAE/MSE/RMSE), анализ ошибок.

---

## 3. Временные ряды и прогнозирование

### 3.1. Прогноз нагрузки на сервис `[INTERMEDIATE]`

**Идея:** прогноз количества запросов/нагрузки на условный сервис.

- Вход: история по часам/дням (число запросов, события, сезонность).
- Выход: прогноз на ближайший период.
- Модели: классические методы временных рядов и простые нейросети/Seq-модели.
- Сервис: API для запроса прогноза.

**Акценты:** работа с сезонностью, валидация на временных рядах.

---

### 3.2. Прогноз продаж или спроса по времени `[BASIC]`

**Идея:** прогноз объёма продаж товара/группы товаров.

- Вход: временной ряд количества продаж.
- Выход: прогноз на следующую неделю/месяц.
- Модели: классические модели временнЫх рядов, базовые нейросети.
- Сервис: минимальный API для получения прогнозов.

**Акценты:** простая, но аккуратная постановка эксперимента и baseline’ы.

---

## 4. Компьютерное зрение (CV)

### 4.1. Классификация изображений в узком домене `[INTERMEDIATE]`

**Идея:** сервис, который определяет класс объекта на картинке (например, тип товара, дефект/без дефекта).

- Вход: изображение.
- Выход: класс и/или вероятность по классам.
- Модели: CNN/transfer learning на готовых backbone’ах.
- Сервис: API, принимающее изображение (либо ссылку/ID) и возвращающее результат.

**Акценты:** аккуратная подготовка данных (аугментации, train/val/test), интерпретация ошибок.

---

### 4.2. Детектирование дефектов или объектов `[ADVANCED]`

**Идея:** сервис, который отмечает объекты/дефекты на изображении.

- Вход: изображение.
- Выход: bounding boxes/маски и классы.
- Модели: готовые detection-модели (YOLO, Faster R-CNN и т.п.) и перенос обучения.
- Сервис: API, которое возвращает координаты и классы объектов.

**Акценты:** сложность аннотаций, работа с предобученными моделями.

---

### 4.3. Классификация «до/после» по картинкам `[INTERMEDIATE]`

**Идея:** различать состояния объекта до/после операции (например, «убрано/не убрано», «собрано/разобрано»).

- Вход: изображение.
- Выход: бинарная классификация или оценка «качества» состояния.
- Модели: CNN с transfer learning.
- Сервис: API для оценки состояния.

**Акценты:** формулировка метрики «качества» и работа с несбалансированными классами.

---

## 5. Текст, NLP и RAG

### 5.1. Классификация обращений/тикетов `[INTERMEDIATE]`

**Идея:** сервис, который по тексту обращения пользователя определяет категорию или приоритет.

- Вход: текст тикета/запроса.
- Выход: категория (или несколько тегов), при необходимости — приоритет.
- Модели: классические текстовые модели и трансформеры (BERT-подобные).
- Сервис: API для классификации.

**Акценты:** предобработка текста, работа с несколькими классами, интерпретация.

---

### 5.2. Семантический поиск по набору документов `[INTERMEDIATE]`

**Идея:** поиск релевантных документов по пользовательскому запросу (без генерации ответов).

- Вход: текстовый запрос.
- Выход: top-N документов/фрагментов с оценкой близости.
- Модели: эмбеддинги текста (трансформеры), векторное пространство.
- Сервис: API, которое возвращает список документов/фрагментов.

**Акценты:** построение индекса, выбор меры сходства, качество поиска.

---

### 5.3. RAG-помощник по локальному корпусу `[ADVANCED]`

**Идея:** помощник, который отвечает на вопросы по заданному набору документов (например, документация, инструкции, FAQ).

- Вход: вопрос пользователя.
- Выход: сгенерированный ответ и ссылки на фрагменты источников.
- Компоненты:
  - индексирование документов;
  - семантический поиск релевантных фрагментов;
  - генерация ответа на основе найденного контекста.
- Сервис: API/чёрный ящик «задал вопрос → получил ответ».

**Акценты:** RAG-пайплайн, качество контекста, ограничения/безопасность ответов.

---

### 5.4. Классификация тональности/тематик отзывов `[BASIC]`

**Идея:** анализ отзывов/комментариев: тональность (positive/neutral/negative) или темы.

- Вход: текст отзыва.
- Выход: тональность и/или тема.
- Модели: от простых (bag-of-words и классический ML) до трансформеров.
- Сервис: API для анализа текста.

**Акценты:** baseline’ы, сравнение простых и сложных моделей.

---

## 6. Рекомендательные системы

### 6.1. Рекомендации товаров/контента `[INTERMEDIATE]`

**Идея:** сервис, который рекомендует товары/контент пользователю.

- Вход: ID пользователя и/или его история.
- Выход: топ-N рекомендованных элементов.
- Модели: простые коллаборативные/контентные рекомендации, гибриды.
- Сервис: API для получения рекомендаций.

**Акценты:** offline-метрики для рекомендаций, холодный старт.

---

### 6.2. Рекомендации похожих объектов `[BASIC]`

**Идея:** «похожие товары/статьи/курсы» по одному объекту.

- Вход: ID объекта.
- Выход: список похожих объектов.
- Модели: эмбеддинги/feature-based similarity.
- Сервис: API для получения списка «похожих».

**Акценты:** понятный feature engineering и визуализация «карты объектов».

---

## 7. Продвинутые мультимодальные и агентные проекты

### 7.1. Мультимодальный поиск: «картинка → текст» или «текст → картинка» `[ADVANCED]`

**Идея:** сервис поиска по картинкам и описаниям.

- Варианты:
  - по тексту найти релевантные картинки;
  - по картинке найти релевантные описания/товары.
- Модели: мультимодальные модели (CLIP-подобные).
- Сервис: API, принимающее запрос и возвращающее список результатов.

**Акценты:** работа с двумя модальностями, интерпретация результатов.

---

### 7.2. Агент-помощник для анализа данных или логов `[ADVANCED]`

**Идея:** агент, который помогает анализировать данные/логи и предлагает next steps.

- Вход: запрос пользователя и доступ к описанным инструментам (например, чтение файла с логами, выполнение простых агрегирующих операций).
- Выход: ответы и краткие пояснения, какие шаги были выполнены.
- Компоненты:
  - orchestration вокруг LLM/модели;
  - интеграция с простыми «инструментами»;
  - логирование шагов агента.

**Акценты:** дизайн интерфейса задач, безопасность исполнения, трассировка.

---

## 8. Как согласовать свою тему

1. Сформулировать идею в 5–10 предложениях:
   - что за задача и кто «пользователь»;
   - какие данные планируются;
   - какой результат/метрика успеха.
2. Прикинуть:
   - какие модели подойдут;
   - какой сервис можно сделать сверху;
   - какие риски (данные, ресурсы, сроки).
3. Написать преподавателю краткое описание:
   - выбрать ближайший к вашей идее домен/пример из этого файла;
   - отметить уровень сложности, к которому вы стремитесь (`[BASIC]`, `[INTERMEDIATE]`, `[ADVANCED]`).
4. После согласования — завести/обновить описание в `project/README.md` своего репозитория и двигаться по этапам, описанным в `project/overview.md`.

Если ваша идея не попадает ни в один из перечисленных шаблонов, это не проблема — важно, чтобы она укладывалась в рамки курса и позволяла пройти полный цикл «данные → модель → сервис → эксплуатация».
