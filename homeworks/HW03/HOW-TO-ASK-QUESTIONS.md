# Как задавать вопросы по курсу

## 1. Зачем этот файл

Этот курс про инженерный подход к ИИ. Инженерам важно уметь:

- формулировать проблему;
- давать контекст;
- задавать конкретные вопросы.

Этот файл — короткий гайд:

- **куда** задавать вопросы;
- **что** писать в вопросе;
- **как** оформить вопрос, чтобы получить полезный ответ.

---

## 2. Где задавать вопросы

Актуальный список каналов уточняется преподавателем, но базовая схема такая:

- **Общий чат курса**  
  Для:
  - уточнений по лекциям и семинарам;
  - вопросов по домашкам, если они полезны всем;
  - организационных моментов (расписание, дедлайны, формат).

- **Личные сообщения / почта / LMS**  
  Для:
  - индивидуальных ситуаций (болезнь, форс-мажор, пересдачи);
  - вопросов, которые нельзя обсуждать публично.

- **Issues / обсуждения в репозиториях** (если используется)  
  Для:
  - багов и предложений по `aie-course-meta`, шаблонам репозиториев;
  - предложений по улучшению материалов.

Конкретные каналы и форматы — в `COURSE-POLICIES.md`. Здесь — как **содержательно** задавать вопросы.

---

## 3. Общие принципы хорошего вопроса

Хороший вопрос по курсу обычно содержит три части:

1. **Контекст** — в какой ситуации вы находитесь:
   - лекция / семинар / домашка / проект;
   - номер лекции/семинара/домашки (`L05`, `S03`, `HW02`, этап проекта).

2. **Что вы уже сделали**:
   - какие шаги прошли;
   - на каком именно месте всё сломалось;
   - что уже пробовали.

3. **Что именно вы хотите получить**:
   - объяснение идеи / термина;
   - подсказку, где искать ошибку;
   - проверку, что формулировка/подход корректны.

Правило:

> Чем меньше собеседнику приходится угадывать за вас, тем быстрее вы получите полезный ответ.

---

## 4. Общий шаблон вопроса

Можно ориентироваться на такой шаблон (адаптируйте под себя):

1. **Тип вопроса**:  
   `Тип: теория / пример с лекции / домашка HWNN / семинар SNN / проект / окружение / git / другое.`

2. **Контекст**:  
   - «Лекция L03, слайд про …»  
   - «Домашка HW02, задание №3, файл …»  
   - «Проект, этап E03: …»  

3. **Что вы делали** (коротко, но по сути):  
   - какие команды/функции вызывали;
   - к какому результату стремились.

4. **Что получилось**:  
   - фактический результат;
   - текст ошибки или непонятный вывод.

5. **Конкретный вопрос**:  
   - «Правильно ли я понимаю, что …?»  
   - «Почему здесь нужно делать X, а не Y?»  
   - «Как корректно исправить эту ошибку в рамках подхода курса?»

---

## 5. Вопросы по содержанию (лекции, семинары, материалы)

Когда спрашиваете по теории или примерам из лекций/семинаров:

- указывайте:
  - номер лекции/семинара (`L07`, `S02`);
  - если есть — пример/слайд/файл (`L07-slides.pdf`, раздел про …).
- формулируйте вопрос так, чтобы было видно:
  - что именно непонятно;
  - как вы это сейчас понимаете.

Примеры формулировок:

- «В лекции L04 на слайде про split данных сказано X. Я понял это так: … Правильно ли, что …?»
- «В демо S03 мы используем такую схему кросс-валидации. Я пытаюсь перенести её в домашку HW03, но не уверен, как…»

---

## 6. Вопросы по домашкам и проекту

Для домашек (`homeworks/`) и проекта (`project/`) желательно указывать:

- **что именно** вы делаете:
  - `HW03`, задание 2;
  - проект, файл `project/src/service/api.py`.
- **где посмотреть код**:
  - ссылка на ваш репозиторий (если это допустимо в формате курса);
  - путь к файлу в репозитории;
  - при необходимости — небольшой фрагмент кода (не весь файл по 300 строк).

Полезно добавлять:

- пример входных данных, на которых проблема проявляется;
- ожидаемый и фактический результат;
- скрин/лог с ошибкой (с читаемым шрифтом и без лишнего шума).

---

## 7. Вопросы по окружению, Git, Python и инструментам

Это самые частые «болезненные» вопросы. Чтобы не тратить время на угадайку, всегда прикладывайте:

- **ОС и способ работы**:
  - Windows / macOS / Linux / WSL;
  - какая IDE (VS Code / PyCharm / другое);
- **версию Python**:  
  `python --version` / `python3 --version`;
- факт, что вы работаете из **корня репозитория**;
- команду, которую вы запускали;
- **полный текст ошибки** (а не только «последнюю строчку»).

Пример структуры вопроса:

- «ОС: Windows 11 + WSL Ubuntu, VS Code. Python 3.10.»  
- «Делаю шаги из `python-101.md` / `unix-101.md`: …»  
- «Запускаю команду: `python -m venv .venv` → получаю ошибку: … (см. цитату ниже)»  
- «Что я уже пробовал: …»  
- «Вопрос: как это корректно починить в рамках рекомендованной конфигурации курса?»

Отдельно полезно указывать, что вы уже заглянули в:

- `git-101.md` — если вопрос про Git;
- `python-101.md` — если про Python/venv;
- `unix-101.md` — если про терминал;
- `jupyter-101.md` — если про ноутбуки;
- `infra-notes.md` / `security-notes.md` — если про сервис и безопасность.

## 8. Примеры «плохих» и «хороших» вопросов

### 8.1. Плохо

- «У меня ничего не работает, помогите»  
- «Git ругается, что делать?»  
- «Ошибка какая-то в Python…»  
- «Сервис не поднимается»  

Эти формулировки не дают точек опоры: не ясно, ни что вы делаете, ни где искать.

### 8.2. Лучше

- «Домашка HW02. Пытаюсь сделать шаг из задания 3: запускаю `python -m venv .venv` в корне репозитория, получаю ошибку `…`. ОС: Windows 11, терминал PowerShell. До этого установил Python 3.11. Вопрос: что я делаю не так и как правильно настроить venv под этот курс?»  
- «Проект, файл `project/src/service/api.py`. В лекции говорилось, что `/predict` должен использовать модель, а не заглушку. Я сделал так: … (фрагмент кода). Вопрос: корректно ли, что я подгружаю модель при старте приложения, или нужно делать иначе?»  
- «Семинар S03, пример с кросс-валидацией. Я пытаюсь выполнить ноутбук `S03-demo.ipynb`, но на ячейке с … получаю `ValueError: …`. Скрин и текст ошибки ниже. До этого шаги 1–3 выполнились без проблем. Вопрос: что я упускаю в подготовке данных?»

Такие вопросы позволяют преподавателю или ассистенту почти сразу дать полезный ответ или навести вас на решение.

## 9. Что сделать перед тем, как задавать вопрос

Небольшой чеклист само-помощи (по возможности):

1. Посмотрите, нет ли ответа:
   - в `COURSE-OVERVIEW.md` / `COURSE-POLICIES.md`;
   - в `homeworks-policy.md` / `seminars-policy.md`;
   - в README текущего репозитория;
   - в `git-101.md`, `python-101.md`, `unix-101.md`, `jupyter-101.md`, `infra-notes.md`, `security-notes.md`.

2. Попробуйте сузить проблему:
   - минимизировать пример;
   - убрать всё лишнее;
   - понять, на каком именно шаге всё ломается.

3. Сформулируйте вопрос так, будто пишете его **коллеге**, который видит задачу впервые:
   - укажите контекст;
   - опишите шаги;
   - приложите факты (сообщения об ошибках, фрагменты кода).

Если после этого вопрос остаётся — смело задавайте. Это не «стыдно», это часть нормальной инженерной работы.

---

## 10. Чего лучше не делать

Чтобы экономить время себе и другим, избегайте:

- сообщений без темы и контекста («помогите», «срочно», «кто жив?»);
- скриншотов ошибок **без текста** (по скрину трудно цитировать и гуглить);
- фото экрана вместо копии текста;
- отправки секретов (пароли, токены, ключи) в чат или письмом;
- обсуждения конкретных решений других студентов без их согласия.

Если нужно показать ошибку:

- скопируйте текст сообщения;
- при необходимости — приложите небольшой фрагмент кода;
- можно добавить скрин, но текст всё равно лучше иметь в вопросе.
